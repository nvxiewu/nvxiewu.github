十一提交更改：

十八更新最后一个提交：
作用显而易见，如果最后一次提交还有需要修改的地方，但是修改的内容又不足以作为一次新的变更提交，那么就需要对原有的提交做修改，为提交命令加 --amend 参数选项即可对最后一次提交做更改：
单独运行git commit --amend只对提交注释进行修改。
对工作目录下的文件做修改，然后加入暂存区再运行git commit --amend提交则只对最后一次提交做修改，不会创建新的提交。

还原提交的意义是撤销某一次提交更改的内容，恢复到某次提交之前的初始状态，然后创建一个新的提交来记录此次还原。

二十重置提交：
重置提交的意义是撤销某一次提交，将其删除或撤销至暂存区或工作目录，并不会对文件的变更做任何改变，其中重置的命令为：
git reset [options] reference-to-commit
其中参数选项[options]包括：--mixed --soft --hard
相关引用reference-to-commit表示方式包括：
HEAD^
HEAD~
HEAD~1 HEAD^^
HEAD~2
HEAD^^^
HEAD~3
HEAD^2
其中相对引用及参数选项的含义请自行百度，这里需要深入了解重置提交的可以查看以下链接：
https://git-scm.com/docs/git-reset
https://git-scm.com/blog
https://git-scm.com/book/zh/v2/Git-工具-选择修订版本
重置提交，特别是采用--hard参数选项后，系统会对清除的提交内容持续跟踪30天，在这期间还可以继续使用git reflog命令调用这些内容，30天以后系统将彻底清除这些提交的内容，想要详细了解git reflog命令可以参考以下链接：
https://git-scm.com/docs/git-reflog

远程仓库学习笔记：
远程仓库的含义就不在此累述，可自行百度，接下来直接进入正题：
一 远程仓库的访问：
方式有两种： 使用URL，使用系统文件的路径
登陆命令： git remote
二 添加远程仓库：
这里以github为例，如何在github上创建一个远程仓库请自行百度，这里以URL作为远程仓库的链接，假设创建的远程仓库名为A，在本地同样创建一个名为A的仓库，将本地仓库和远程仓库做连接的命令如下：
git remote add X URL
其中X为远程仓库的简写名，通常为origin，也可以是其它任意简写名，用于代替URL。
采用命令git remote 用于检验远程仓库，加-v参数则可以查看远程仓库和链接的详细信息，想更深一步了解远程仓库可以参考以下链接：
https://git-scm.com/docs/git-remote
https://git-scm.com/book/en/v2/Git-Basics-Working-with-Remotes#_showing_your_remotes
三 将更新推送至远程仓库：
推送至远程仓库的命令：
git push remote-shortname branch
其中remote-shortname是远程仓库的简写，branch是要推送的分支
推送后在本地使用git log查看可以看到HEAD除了指向本地仓库的当前分支外，后面还有一个被跟踪的remote-shortname/branch的远程仓库分支称为跟踪分支
四 从远程仓库拉取更改：
拉取远程仓库的命令：git pull remote-shortname branch
拉取远程仓库的命令使得远程仓库会的更改被复制到本地仓库，本地跟踪分支会快进合并至本地分支。
特别的采用： git fetch remote-shoutname branch会将远程仓库的更改复制到本地仓库，但是和采用git pull不同的是其不会快进合并至本地分支。

以下是贡献者的最佳实践：
1.编写描述性的提交说明 在谈论如何命名分支，以清晰描述分支会包含哪些更改的同时，我想另外提醒一下如何编写清晰、描述性的提交说明。你的分支名称和提交说明描述得越清楚，项目维护者用于询问你的代码的用途，或者自己去深入了解代码的时间就越少。项目维护者需要做的工作越少，将你的更改纳入项目的速度就越快。
2.创建短小而明确的提交
这一点我们之前已经强调了很多次，请确保在对项目 commit 更改时，使用短小的 commit。不要进行大量 commit，记录 10 多个文件和数百行代码的更改。最好频繁多次地进行小的 commit，只记录很少数量的文件和代码更改。
3.更新README
最后，如果你添加的任何代码更改会使项目发生极大的变化，则应更新 README 文件以向其他人说明此更改。

九 与源项目保持同步：
首先对于想保持关注和跟进的项目可以在github上使用Star和Watch收藏并关注项目，具体方式可以自行百度。 其次如果想本地仓库和源仓库保持同步则可以采取以下步骤：
假设已经fork了源仓库并克隆至本地计算机创建了特性分支做更改提交后，如果源仓库已经变更则：
获得源仓库的可克隆URL
使用git remote add命令创建一个新的远程仓库
使用简写名upstream指向源仓库
提供源仓库的URL
使用git fetch upstream master命令拉取源仓库在master分支上的更改 采用git merge命令将upstream仓库更新的跟踪分支合并到本地分支
将新更新的本地分支推送至origin仓库
注意这里为了避免混淆，可以使用git remote rename命令对远程仓库进行重命名，语法格式如下：
git remote rename mine origin

这里管理活跃PR的含义是当提交PR后可能项目的运营者会要求你将提交结合在一起后才将其合并，接下来你需要在本地仓库的统一分支上添加一些必要的提交然后将该分支推送至源仓库的fork副本，这些提交最终将显示在Pull Request页面上。
十一压制提交：
这里压制提交有很大的现实意义，当你发现有一系列的提交都有相同的特性，则可以将其压制为一个提交，这样整个提交历史将更为简洁，其次在PR的过程中，运营者也可能以相同的理由要求你对PR的提交进行压制，以下为压制的命令：
git rebase -i base
这里-i参数代表“交互式”，默认情况下建议加-i参数；base代表相关引用，通过相关引用指向一个祖先提交作为提交压制的基提交，称为变基，以下为压制变基的一个常用流程：
1.创建一个beckup分支指向最近的提交（目的显而易见，变基将删除被压制的提交，创建beckup分支可以将其备份保留，以便在变基不满意的情况下还能恢复到原有的提交）
2.使用压制命令指定一个基底祖先引用，记得采用-i参数（这里作为基底引用除了可以采用相对引用以外还可以使用SHA，分支名称，标签名称）
3.压制开始后将进入编辑器指定rebase命令：
p或pick - 使commit保持原样
r或reword - 保留commit的内容，但修改commit说明
e或edit - 保留commit的内容，但先不执行commit，以便：
添加新内容或文件
删除内容或文件
修改即将commit的内容
s或squash - 将此commit的更改结合到前一个commit中
f或fixup - 将此commit的更改结合到前一个commit中，但删除提交说明
x或exec - 运行shell命令
d或drop - 删除commit
（这里对如何使用验证命令不清楚的可以自行百度）
4.如果提交了PR，则可以将压制推送至fork副本，PR将自动更新，但是推送过程中会被github拒绝，因为就像前面所述，压制命令将会删除被压制的提交，这会被github视为一种威胁而拒绝，这实际上是一种帮助，确保你已经审查了正在推送的提交，这时可以采用强制推送参数将压制后的提交推送至fork副本：
git push -f
5.对于何时适合进行变基的情况，一般如果项目在与其他开发者协作，那么推送后则不应该变基，因为如果强制推送则可能造成其他开发者无法和远程仓库同步，需要对Git仓库进行复杂的手术，甚至于不得不抛弃之前的所有工作。

查询的含义是，按照一定的条件查找数据库中所需要的信息，一条典型的数据库查询命令如下：
select food from diet where species = 'orangutan'
在实际工作中，我们通过数据库查询命令来获得所需要的数据，通常工作模式和数据库的部署相关，一般来讲数据库部署可以是本地也可以是远程，比如的常见的独立部署的数据库包括mysql,postgresql,oracle等，常见的本地部署的数据库有sqlite，如果是部署远程数据库，那么查询命令通过网络传输到远程数据库服务器，再通过数据库服务器读取数据物理盘来获得结果，而本地部署的数据库则通过查询命令直接调用DB库来读取物理盘获得结果。
忽略掉具体部署模式，我们以刚才举例的查询命令为例
select food from diet where species = 'orangutan'
这条查询命令将返回一个表，这个表包含了一列两行，列名就是要查询的food列，结果就是表中所有猩猩喜欢吃的食物的列表。
另我们还可以用select 2+2这样的命令获得一张结果表，但是这张表只有一行一列，并且列名为特殊字符，也可以用select 2+2，3+3，4+4这样的命令获得一张三列一行构成的结果表，同时也可以用select 2+2 as sum这样的命令指定结果表的列名

select products.name,products.sku,count(*) as num  
   from products join sales

上面这种连接方式将无法统计出products表中每件商品出现在sales表中的次数，因为如果某件商品没有被售卖过，那么将无法在结果中呈现出来，所以我们需要采用左连接：  
select products.name,products.sku,count(sales.sku) as num

    group by products.sku;  
以上连接表示左侧表格是products，右侧是sales，该连接将返回左侧表格中的所有行，而普通的inner连接只返回两个表格中符合连接条件的行，关于左右连接的更多知识可以自行百度查询。

(6)迭代器和生成器：
迭代器是每次可以返回一个对象元素的对象，例如返回一个列表，列表是一个可迭代对象，但不是迭代器，迭代器是一种表示数据流的对象，生成器是使用函数创建迭代器的简单方法。也可以使用类定义迭代器，更多详情可以参考以下链接：
https://docs.python.org/3/tutorial/classes.html#iterators
以下是一个生成器函数示例：
def my_range(x):
i=0 while i<x:
yield i
i+=1
使用该函数创建的迭代器：
for x in my_range(5): print(x)
输出结果为：0 1 2 3 4
为什么要使用生成器，而不直接使用可迭代对象。以下两个链接是可能的参考解释：
https://softwareengineering.stackexchange.com/questions/290231/when-should-i-use-a-generator-and-when-a-list-in-python/290235
https://softwareengineering.stackexchange.com/questions/273551/should-i-prefer-python-generators-to-lists
可以认为生成器是构建迭代器的“懒惰”方式。当内存不够存储完整实现的可迭代对象时，可以使用生成器。但是生成器产生的元素只能遍历一次而不能回溯，以下是一个生成器实际使用的一个示例：
def chunker(iterable,size):
for i in range(0,len(iterable),size):
yield iterable[i:i+size]
for chunk in chunker(range(25),4):
print(list(chunk))

三 URI
具体概念这里不累述，不清楚的可以百度，这里URI是Uniform Resource identifier的缩写，URL和URI非常像，其主要应用于互联网资源，是Uniform Resource Locator的缩写，但是URI的描述更为精确
一个简单的URI示例如下：
https://en.wikipedia.org/wiki/Fish
这里有三个组成部分：
https表示方案scheme
en.wikipedia.org表示主机名hostname
/wiki/Fish表示路径
下面就每个部分进行分开介绍：
1.Scheme(方案)
这里的方案是指客户端以什么方式获取资源，例如http,https和file，其中File表示客户端以本地文件系统方式获取资源，http和https表示以web服务的方式提供资源，更多方案列表请参考以下链接：
http://www.iana.org/assignments/uri-schemes/uri-schemes.xhtml
注：(其中http和https的区别是，https的文件传输经过了加密连接，而http则没有加密。另外客户端不仅仅指浏览器，还包括例如app，小程序等客户端)
2.主机名
顾名思义，主机名是用于确定客户端需要连接的资源提供者，主机名通常需要紧跟方案，并用://连接两方，例如上面例子中的www.iana.org。
注：主机名在URI中并非必选项，有些方案后面不需要紧跟主机名，例如mailto:spam@example.net，方案mailto后面以冒号开始，并跟了一个邮件地址，这种情况下也就不需要//符号。
3.路径Path
路径用于确定主机上资源的具体位置，例如不同的页面，视频或者API，这些path用于告诉客户端所需资源的具体位置，例如上面例子中的/assignments/uri-schemes/uri-schemes.xhtml 注：当URI后面不跟路径，客户端将自动填充一个斜杠，指向主机资源的根目录，例如http://udacity.com，浏览器将自动填充为http://udacity.com/
4.查询组件
例如：
https://www.google.com/search?q=fish
其中?q=fish为查询组件，并将该查询参数传递给服务器
5.fragment组件
例如：
https://en.wikipedia.org/wiki/Oxygen#Discovery
其中#Discovery部分为fragment组件，该组件一般对应到HTML的id元素，用于客户端连接至对应的资源片段。
6.相对URI
例如在HTML文档内的<a>标签内的链接：
<a href='cliffsofinsanity.png'>cliffsofinsanity.png</a> 其中href='cliffsofinsanity.png'为相对URI，链接到资源目录内的cliffsofinsanity.png图片。
总之一个标准的URI语法一般由以下五个分层序列组成：
URI=scheme:[//权限]路径[?查询][#fragie]
其中中括号内的为可选组件，更多URI的详细信息可以参考以下链接：
https://en.wikipedia.org/wiki/Uniform_Resource_Identifier
四 Hostnames and ports
这里着重介绍一些概念：
1.主机名对应着互联网上的服务器，但是并非直接对应，而是通过IP地址对应，所以主机名一般还需要通过DNS服务的查询，返回服务器的准确IP地址再请求资源。
2.在终端通过host或nslookup命令查询主机名的IP地址，例如：
host www.google.com
www.google.com has address 216.58.194.164
3.端口Ports对应了主机中服务程序的位置，但是并非所有主机后面都会跟端口，原因是有些端口对于有些方案是默认的，例如HTTP默认端口是80，而HTTPS默认端口是443，在这种情况下URI中的主机名后就不用跟端口号，例如：
https://en.wikipedia.org/wiki/Fish等同于
https://en.wikipedia.org:443/wiki/Fish
通常1024以下的端口号都默认被用于系统端口，自定义端口号通常需要大于该数值，并且需要在URI中显示的调用。
注：所有通过网络传输的数据都将被拆分为包，而包中就需要包含发送方和接收方的IP和端口号。

八 encode()编码
例如在第七部分添加response body时，使用了encode()方法：
self.wfile.write('Hello, HTTP!\n'.encode())
其原因是为了统一各类文本信息在互联网的传输，需要对文本先进行转码，转为字节对象，然后再传输，encode()方法就是将普通文本转码为字节流对象，而decode()方法则与之相反，是将字节流对象转码为普通文本(strings)对象，其原因主要是因为不同文字所要占据的二进制位不同，例如英文单词一个字母就占据一个byte，而一个日语文字或汉字有可能占据多个byte，如果直接以string形式在网络上传输，英文以外的文字将无法被很好的传输和解析，即出现所谓的乱码，为此需要在传输之前对文本信息进行转码并在收到信息后再解码，而目前最常用的编码方式就是UTF-8，该编码方式可以应用于所有文字的传输，需要更详细的了解该部分内容可以参考以下链接：
https://docs.python.org/3.6/howto/unicode.html

十二GET and POST
首先，关于GET和POST的具体区别可以自行百度，这里只简单总结一下：首先GET方法从form提交的参数都会在URI上面显式的传递给服务器，GET方式比较适合用于查询，或者向服务器请求某种资源，而不适合去修改或者创建一个新的资源，如果要修改或者创建一个新资源，适合使用另外一种方式就是POST方式去处理，这里面涉及到一个概念，即幂等性，幂等性是某一动作运行两次或以上次数得到的结果和运行一次的结果一样，例如查询“偶数的定义”，无论查询几次，其结果都是一样的(假设查询过程中没有其它因素改变预定的答案)，查询本身不会改变查询结果；而非幂等性是指某一动作每次运行都会修改结果，比如向购物车添加一罐啤酒，没运行一次，购物车里面的啤酒数量都会增加，而GET方法是幂等性的，POST方法是非幂等的。

二二Cookies
关于Cookies的概念就不在这里累述了，这里只需要清楚Cookies是由服务器设置，由浏览器保存，并且在随后的查询中由浏览器再回传至服务器的一种机制，Cookie将被应用于HTTP的一些高级特性，例如sessions或login等。
1.Cookie的创建：
首先客户端向服务端发送request请求，服务端回传一个包含Set-Cookie组件header的response，这个header包含三方面的内容: cookie name,a value,and some attributes.随后所有的查询都将包含这些信息的cookie回传至服务端，服务端可以更新这些cookie。
2.在本地浏览器查看cookies:
方式可以自行百度，这里需要说明的是，在本地修改cookie可能导致服务器端的一些预期行为受到影响，我们以本地的一个cookie示例来说明其包含的信息：
Name: _ga
Content: GA1.2.257731326.1479515271
Domain: .edhrec.com
Path: /
Send for: Any kind of connection
Accessible to script: yes
Created: Friday,November 18,2016 at 4:27:50 PM
Expires: Sunday,November 18,2018 at 4:27:50 PM 

(1)头两项Name和Content，也被称为cookie的key和value，其中key的命名由一些特殊的规则约束，例如不能查询空格；value是表示cookie的一些真实的数据，例如表示已经登陆用户的唯一令牌  
(2)后面两项是Domain和path，用于描述cookie的作用域，表示包含了哪些查询，默认情况下，domain来自于URI中的主机名。 (3)后面两项'Send for'and'Accessible to script'在内部被称为Secure和HttpOnly，其值为布尔值，分别用于设置是否只是接受HTTPS连接或者是否接受JavaScript在网页端运行。  
(4)最后两项Created和Expires用于确定cookie的创建和失效时间，其中Expires的设置也等同于Max-Age域的设置，如果不设置Expires或者Max-Age，则表示cookie在浏览器关闭时失效。  
3.在python中使用cookie

# An HTTP server that remembers your name (in a cookie)

# In this exercise, you'll create and read a cookie to remember the name
# that the user submits in a form.  There are two things for you to do here:
#
# 1. Set the relevant fields of the cookie:  its value, domain, and max-age.
#
# 2. Read the cookie value into a variable.

from http.server import HTTPServer, BaseHTTPRequestHandler
from http import cookies
from urllib.parse import parse_qs
from html import escape as html_escape

form = '''&lt;!DOCTYPE html&gt;
    &lt;title&gt;I Remember You&lt;/title&gt;
    &lt;p&gt;
    {}
    &lt;p&gt;
    &lt;form method="POST"&gt;
        &lt;label&gt;What's your name again?
        &lt;input type="text" name="yourname"&gt;
    &lt;/label&gt;
    &lt;br&gt;
    &lt;button type="submit"&gt;Tell me!&lt;/button&gt;
    &lt;/form&gt;
'''


class NameHandler(BaseHTTPRequestHandler):
    def do_POST(self):
        # How long was the post data?
        length = int(self.headers.get('Content-length', 0))

        # Read and parse the post data
        data = self.rfile.read(length).decode()
        yourname = parse_qs(data)["yourname"][0]

        # Create cookie.
        c = cookies.SimpleCookie()

        # 1. Set the fields of the cookie.
        #    Give the cookie a value from the 'yourname' variable,
        #    a domain (localhost), and a max-age.
        c['yourname']=yourname
        c['yourname']['domain']='localhost'
        c['yourname']['max-age']=600
        # Send a 303 back to the root page, with a cookie!
        self.send_response(303)  # redirect via GET
        self.send_header('Location', '/')
        self.send_header('Set-Cookie', c['yourname'].OutputString())
        self.end_headers()

    def do_GET(self):
        # Default message if we don't know a name.
        message = "I don't know you yet!"

        # Look for a cookie in the request.
        if 'cookie' in self.headers:
            try:
                # 2. Extract and decode the cookie.
                #    Get the cookie from the headers and extract its value
                #    into a variable called 'name'.
                in_c=cookies.SimpleCookie(self.headers['Cookie'])
                name=in_c['yourname'].value
                # Craft a message, escaping any HTML special chars in name.
                message = "Hey there, " + html_escape(name)
            except (KeyError, cookies.CookieError) as e:
                message = "I'm not sure who you are!"
                print(e)

        # First, send a 200 OK response.
        self.send_response(200)

        # Then send headers.
        self.send_header('Content-type', 'text/html; charset=utf-8')
        self.end_headers()

        # Send the form with the message in it.
        mesg = form.format(message)
        self.wfile.write(mesg.encode())


if __name__ == '__main__':
    server_address = ('', 8000)
    httpd = HTTPServer(server_address, NameHandler)
    httpd.serve_forever()
'''

二三DNS和cookie安全
这里主要注意当DNS服务器被入侵后，其主机名对应的IP地址可能被篡改，造成客户端访问到虚假的服务器，客户可能获取用户的用户名和密码，从而造成安全隐患。
二四HTTPS
HTTPS相对于HTTP指服务器和客户端以加密形式连接，其采用的标准加密协议称为TLS(https://en.wikipedia.org/wiki/Transport_Layer_Security)，其主要提供以下保护：
1.所有客户端和服务器端的数据都经过加密传输，其数据只能由该客户端及服务器端读取。
2.使客户端可以认证服务器端，例如当客户端访问www.udacity.com，客户端可以确定服务器发回的response确实来自于udacity的真实服务器。
3.保护客户端及服务端数据传输的完整性，确保其在传输过程中不出现意外或者被篡改。
注：TSL协议类似于更早的SSL协议，可以把SSL看作是TSL的早期版本。

二五TLS检查
要在本地计算机查看TLS可以参考以下链接：
Chrome:https://support.google.com/chrome/answer/95617
Firefox:https://support.mozilla.org/en-US/kb/secure-website-certificate
Safari:https://support.apple.com/zh-cn/guide/safari/sfri40697/mac

1.私钥及证书：
TLS协议中的服务端配置了两个重要的数据：私钥及证书，私钥一直保存在服务器端，而证书将发送至每个连接的客户端，而每个证书中又包含了公钥，私钥和公钥之间匹配了类似暗号的数据加密算法，使得两者之间的数据加密成为可能。 而证书的发放是由证书认证机构CA进行发放，其功能类似于文书公证，证明其申明的主机名及IP地址的真实有效性。 关于如何通过公钥-私钥进行数据加密解密确保数据安全的详细信息可以参考以下链接：
https://en.wikipedia.org/wiki/Public-key_cryptography
2.TLS如何实现认证：
证书的作用就是保障其声明的某个服务，比如google.com其对应的资源不会错误的指向其它机构或者组织，当你获得了google.com发送的证书后，其中包含了DNS域的官方认证信息，一旦客户端访问的主机名在DNS解析后的IP与证书中注册的地址不一致的时候，客户端将发出严重警告，确保客户端不会访问到虚假的服务。
3.TLS如何确保数据完整性：
TLS通过MAC机制确保传输数据的完整性，详细原理可以参考以下链接：
https://en.wikipedia.org/wiki/Message_authentication_code

二六其它方法
不同的HTTP方法都定义了其不同的用法，比如HTML中的submitting form总是由GET或者POST进行处理，但是在HTTP中，还有大量的数据是通过API的形式通过客户端应用进行处理，比如用JavaScript去访问这些API，所以这里就简单介绍一下HTTP中的其它方法的应用(主要用于API，供JavaSctipt直接访问)：
1.PUT方法：
该方法用于创建一个新的资源，其结果返回201状态，该方法成功后请求相同的URI将返回新创建的资源。
注：PUT也可以用于更新资源，但是另外一个不通过JavaSctipt更新资源的方式是用POST方法，通过HTML更新。
2.DELETE方法：
顾名思义，该方法用于删除资源。
3.PATCH方法：
PATCH方法是一个新加入HTTP的方法，类似于git中的修补patching概念，对已经创建的资源进行修补，对其中如何组织和传输数据格式没有标准，一个常见的数据格式为JSON Patch，另一个是JSON Merge Patch，具体可以查看以下链接： http://jsonpatch.com/
https://tools.ietf.org/html/rfc7386
4.HEAD,OPTIONS,TRACE方法：
这三个方法主要用于调试，其中HEAD方法类似于GET，但是只返回headers组件，不返回content；OPTIONS主要用于找出服务器支持的特性；TRACE方法用于回显服务器收到的客户端内容，但是因为安全原因经常被禁用。 

总之，HTTP的每个方法都有其规定的标准使用范围及方法，但是这些都不是强制性的，你可以在实际中不按照标准定义其行为，例如在GET方法中删除资源，但是这样可能造成意想不到的结果，所以一般来讲都需要按照其定义的标准方法来使用，更多详细信息可以参考以下链接：  
https://www.w3.org/Protocols/rfc2616/rfc2616-sec9.html

2.多链接：
HTTP/1.1一般最多一次可以保持6个连接，但是同一时间只能有一个连接获取资源，其它连接只能等待，这造成了延迟，而HTTP/2则打破了这个限制，在同一时间可以发送多个连接，并且每个连接都可以在第一时间获得服务器的资源，这造成了HTTP/2的用户体验要远好于HTTP/1.1。 3.服务推送
HTTP/2还支持主动推送，如果客户端请求了某个资源，并且服务器知道它还会请求相关其它资源，则服务器不会等待客户端再次请求再发送，而是在第一次请求后就主动推送相关资源。
